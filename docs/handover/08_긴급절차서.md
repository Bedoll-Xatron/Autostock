# 08. 긴급 절차서 (Incident Response)

**용도:** 작업 중 또는 운영 중 사고 발생 시 즉시 참조. 6주 작업 기간뿐 아니라 **상시 보존**.

---

## 0. 가장 중요한 한 가지

> 🚨 **모르겠으면 일단 멈춥니다. 그리고 사용자께 보고합니다.**
>
> 잘못된 조치로 손실이 확대되는 것보다, 1시간 멈춤으로 매수 기회를 놓치는 게 100배 낫습니다.

## 1. 킬스위치 (Kill Switch) — 즉시 정지

### 1.1 가장 빠른 정지 (10초 이내)

**모든 매매 활동 즉시 중단:**

```powershell
# logs/.stop 파일 생성 → daemon 재시작 루프 차단
# (현재 코드에 이미 구현됨 — main.py 참조)
New-Item -Path "D:\INFORUN\chutzrit\autostock\logs\.stop" -ItemType File -Force

# 그 다음 서버 프로세스 종료
Get-Process python | Where-Object { $_.MainWindowTitle -match "autostock" } | Stop-Process -Force

# 또는 PID 직접
Stop-Process -Id <PID> -Force
```

### 1.2 매수만 정지 (보유 포지션은 감시 유지)

**파일:** `.env`

```
TRADING_PAUSED=true
```

이후 `executor.execute_decisions` 진입부에서 다음을 검사하도록 **W1 작업 시 미리 추가**해두십시오:

```python
import os
if os.environ.get("TRADING_PAUSED", "false").lower() == "true":
    log.warning("TRADING_PAUSED=true — 매수 중단")
    bot_ui.schedule_message("⛔ TRADING_PAUSED 활성화 — 신규 매수 중단")
    return []
```

> 💡 이 토글은 W1 작업 시 **반드시 같이 구현**하십시오. 사고 발생 후 추가하면 늦습니다.

### 1.3 보유 포지션 일괄 청산

```powershell
# 모든 보유 종목 시장가 매도 (긴급 시에만)
python -c "from autostock.trading.kis_client import get_all_holdings, market_sell; [market_sell(h['ticker'], h['qty']) for h in get_all_holdings()]"
```

> ⚠️ **이 명령은 모든 포지션을 시장가로 청산합니다.** 갭다운 상황에서 실행하면 손실이 확정됩니다. 사용자 승인 없이 절대 실행 금지.

## 2. 사고 시나리오별 대응

### 2.1 시스템이 같은 종목을 반복 매수

**증상:** 같은 ticker로 BUY 알림이 분 단위로 발생

**원인 후보:**
- HITL 자동승인이 무한 루프
- thread_id 충돌

**즉시 조치:**
1. `TRADING_PAUSED=true` 설정
2. 서버 재시작
3. `decisions` 테이블에서 중복 매수 건 확인
4. KIS 잔고와 DB의 `held_positions` 대조

### 2.2 손절선이 깨졌는데 매도 안 됨

**증상:** 텔레그램 손절 알림 없음, 가격은 stop_price 한참 아래

**원인 후보:**
- KIS API 인증 만료
- `_sell_with_retry`가 3회 모두 실패
- trailing_stop 감시 루프가 죽음

**즉시 조치:**
1. `logs/autostock.log` 마지막 100줄 확인
2. KIS 인증: `python -c "from autostock.trading.kis_client import _refresh_token; _refresh_token()"`
3. 수동 매도: KIS 모바일 앱 또는 `manual_buy.py` 의 매도 함수
4. 그 다음 시스템 재시작

### 2.3 텔레그램 알림 안 옴

**증상:** 시장은 움직이는데 알림 0건

**원인 후보:**
- TELEGRAM_BOT_TOKEN 만료/오류
- 텔레그램 API 일시 장애
- `bot_ui.schedule_message` 큐가 막힘

**즉시 조치:**
1. `curl https://api.telegram.org/bot<TOKEN>/getMe` 로 봇 살아있는지 확인
2. `logs/autostock.log` 에서 "telegram" 키워드 검색
3. 알림 없어도 시스템은 동작 중일 수 있음 — **DB와 KIS 잔고를 직접 확인**

### 2.4 모의 → 실거래 전환 사고 (KIS_SIMULATED_MODE=false 잘못 설정)

**증상:** 알 수 없는 종목 매수 알림, 거액 주문

**즉시 조치:**
1. **`logs/.stop` 파일 즉시 생성**
2. KIS 모바일 앱에서 모든 미체결 주문 취소
3. **사용자께 즉시 보고**
4. .env 원복 후 재시작

> ⚠️ **본 시스템은 모의투자 전용으로 운영 중입니다.** 후배가 단독으로 KIS_SIMULATED_MODE를 false로 변경하는 것은 **사고**입니다. 작업 중 .env 변경 시 반드시 diff 확인.

### 2.5 갭다운 직전, 보유 종목 5개 모두 stop 깨질 위기

**상황:** 야간 미국 -3%, 시간외 한국 -5%, 09:00 갭다운 예상

**조치 옵션 (사용자 승인 필수):**

**Option A — 수동 일괄 청산 (08:50)**
- 시간외 종가에서 매도. 갭다운 폭은 줄지만 야간 호가 슬리피지 감수.

**Option B — 갭다운 회피 가드 의존 (W3 작업 후)**
- 09:00~09:05 갭다운 → 5분 대기 → 회복 시 보유 / 미회복 시 매도
- 가장 risk-on 옵션

**Option C — 전부 보유 (Stay)**
- 갭이 일시적이라 판단 시. 통계적 권장 안 함.

> 💡 **사용자께 옵션 A/B/C 제시 후 결정 위임**. 후배 단독 결정 금지.

## 3. 롤백 절차 (코드)

### 3.1 직전 1주차만 롤백

```powershell
# git revert로 안전하게
git log --oneline | Select-Object -First 10
git revert <commit_hash>
python main.py --restart
```

### 3.2 6주 전체 롤백 (최후)

```powershell
git log --oneline --grep "갭업 진입 차단"
git reset --hard <W1 직전 commit>
python main.py --restart
```

> ⚠️ `git reset --hard`는 **사용자 승인 필수**. 매주 회고 후 **git tag** 를 찍어두면 안전:
>
> ```powershell
> git tag w1-baseline
> git tag w2-baseline
> # 이후 롤백 시:
> git checkout w2-baseline
> ```

## 4. 사용자 보고 양식 (사고 발생 시)

```
[사고 보고] {YYYY-MM-DD HH:MM}

상황: (1줄)
영향: (보유 종목, 잠정 손실액)
즉시 조치: (이미 취한 행동)
원인 추정: (가능한 한)
다음 단계 옵션:
  A. ___
  B. ___
  C. ___

후배 추천: B (이유 1줄)

결정 요청드립니다.
```

이 양식대로 텔레그램 또는 직접 사용자께 보고. **추측을 단정으로 보고하지 마십시오** — "원인 추정" / "잠정"이라는 단어를 쓰는 게 정직한 보고입니다.

## 5. 사고 회고 (Post-Mortem) 양식

사고 종료 후 24시간 내 작성. `docs/handover/incidents/` 에 저장.

```markdown
# 사고 회고 — 2026-MM-DD

## 타임라인
- HH:MM 사고 인지
- HH:MM 1차 조치
- HH:MM 사용자 보고
- HH:MM 정상화

## 영향
- 잠정 손실: __원
- 영향 종목: __
- 영향 시간: __분

## 근본 원인
(코드 위치 또는 설정)

## 즉시 조치
(이번에 한 일)

## 항구 조치 (Action Items)
- [ ] 코드 보강: ___
- [ ] 테스트 추가: ___
- [ ] 모니터링 추가: ___
- [ ] 문서 보강: ___

## 책임 (No-blame)
- 사고 자체에 누구를 비난하지 않습니다. **시스템이 사고를 허용한 것이 문제**입니다.
- 단, **보고 지연이나 무단 조치**는 별도 평가 사유입니다.
```

## 6. 책임자 마지막 당부

> "이 절차서를 한 번도 안 펴봤다면 좋은 일입니다. 한 번이라도 폈다면 — **다음 주에 또 펼 일이 없도록** 사후 작업을 끝내십시오."

> "사고는 항상 **'우리 시스템에서 일어날 일이 없는 일'**이라고 생각하던 자리에서 일어납니다. 매주 회고 시 '이번 주에 우리 시스템에서 절대 일어날 수 없는 일은 무엇인가?'를 한 줄 써 보십시오. 그게 다음 사고의 후보입니다."

> "마지막으로 — 이 시스템은 **사용자의 자산을 다루는 코드**입니다. 모의투자라 해도 다음 단계는 실거래입니다. 매 commit, 매 .env 수정, 매 배포가 그 자산에 영향을 줄 수 있다는 자세를 잃지 마십시오."

— 책임자
