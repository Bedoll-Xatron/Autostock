# 09. 점수 임계값 검토 가이드 (이벤트 트리거 문서)

**용도:** 점수 임계값(scorer 등급, supervisor buy_threshold, Bear 점수 기준)을 만져야 할 시점이 왔을 때 따르는 절차서.
**위치:** 사용자 룰 §1의 "임시방편 임계값 변경 금지"의 운영 가이드.
**호출 시점:** 정해진 일정이 아닌 **이벤트 트리거** — 아래 §3 발동 조건 중 하나라도 충족 시.

---

## 0. 이 문서가 존재하는 이유

사용자께서 명확히 하셨습니다.

> "기본률 유지도 코드의 샘플을 보여주기식으로 기준을 자꾸 바꾸는 것 같아 방지를 요청한 사항으로 **문제가 있다면 관리자께서 과감하게 수정해주세요.**"

즉 "기준값 변경 금지" 룰의 진정한 목적은:
- **금지하는 것:** 즉흥적·결과맞추기식 임계값 조작 (땜질 코드)
- **허용하는 것:** 근본 원인 분석 + 측정 데이터 + 책임자 결단을 동반한 변경

본 문서는 **후자의 변경을 안전하게 수행하는 절차**입니다.

---

## 1. 우리 시스템의 점수 임계값 인벤토리

각 임계값과 위치, 그리고 그것이 무엇을 의미하는지 후배가 알아야 만질 수 있습니다.

### 1.1 Scorer 등급 임계값

**파일:** `marketflow/engine/scorer.py:359-370`

```python
def determine_grade(self, stock, score):
    if not score.mandatory_passed:
        if score.vcp_score > 0:
            return Grade.W
        return Grade.C
    if score.total >= 11:    # ⭐ A등급 임계값
        return Grade.A
    if score.total >= 9:     # ⭐ B등급 임계값
        return Grade.B
    if score.vcp_score > 0:
        return Grade.W
    return Grade.C
```

| 임계값 | 현재 | 의미 | 영향 |
|---|---|---|---|
| Grade A | 11점 | A등급 (R배수 2.0배) | 큰 포지션 |
| Grade B | 9점 | B등급 (R배수 1.0배) | 표준 포지션 |
| Grade W | VCP > 0 | 워치(미진입) | 모니터링 |

**`config.py:31` 의 `min_total_score=9` 도 같이 검토 대상.**

---

### 1.2 Supervisor buy_threshold

**파일:** `autostock/research/supervisor.py:51`

```python
buy_threshold = 5 if config.KIS_SIMULATED_MODE else 7
```

| 모드 | 현재 | 의미 |
|---|---|---|
| 모의 (KIS_SIMULATED_MODE=true) | **5점** | 의도된 공격성 — 백테스트 데이터 수집 목적 |
| 실거래 (false) | **7점** | 보수적 진입 |

> ⚠️ **사용자께서 명시한 의도:** "모의투자라 공격적으로 구매해 보려고 하는 의도였으니 필요 시 수정 반영." 즉 모의 5점은 의도된 정책. 손절률이 너무 높다는 이유만으로 5→6으로 올리는 건 **사용자 의도와 충돌**할 수 있음. 변경 전 §3 절차로 사용자 확인 필수.

---

### 1.3 Bear 점수 캘리브레이션

**파일:** `autostock/research/debate.py:67`

```python
"실질적인 리스크가 없다면 낮은 bear_score(1~3)를 부여하세요. "
"치명적 리스크(상장폐지 위험, 횡령, 거품 붕괴 임박 등)에만 높은 점수(8~10)를 부여하세요."
```

| 구간 | 의미 | 효과 |
|---|---|---|
| 1~3 | 일반 리스크 (현재 거의 모든 종목이 여기 떨어짐) | Bear 영향력 미미 |
| 8~10 | 치명적 리스크 | CIO가 HOLD 결정 |
| **4~7 (현재 빈 영역)** | **중간 리스크가 표현될 자리** | 이 구간이 비어있어 Bear가 사실상 침묵 |

**현 상태:** Bear는 8점 이상 안 주면 영향이 없음 → CIO는 거의 항상 BUY/HOLD 중 BUY 쪽으로 기울게 됨. 모의 5점 임계값과 결합되면 매우 공격적 매수 시스템이 됨. 사용자의 "공격적 매수 의도"와는 일치하지만, **W1~W4 작업으로도 손절률이 안 떨어지면 이 구간을 정상화할 필요**.

### 1.4 confidence 기반 R 배수

**파일:** `autostock/trading/executor.py:13`

```python
_R_MULTIPLIER_TIERS: list[tuple[float, float]] = [
    (9.0, 1.5),   # confidence ≥ 9 → 1.5배
    (7.0, 1.0),   # confidence ≥ 7 → 1.0배
    (5.0, 0.5),   # confidence ≥ 5 → 0.5배
]
```

이 표는 buy_threshold와 별개로 **포지션 크기**를 결정. 모의 5점 매수의 비중은 0.5배로 이미 보수적임. **즉 사용자 의도(공격적 매수)는 점수 컷오프에서 표현되고, 실제 포지션 크기는 자동 축소되는 균형 구조**임.

## 2. 임계값 변경의 두 종류

### 2.1 ❌ 금지된 변경 (사용자 룰 위반)

다음 패턴은 절대 금지:
- "오늘 손절률이 높네 → A등급 11점을 12점으로 올리자"
- "매수가 너무 적다 → buy_threshold 5점을 4점으로 내리자"
- "Bear가 자꾸 BUY를 막네 → bear_score 임계값을 8에서 9로 올리자"
- (공통점: 측정 1주 미만, 근거 1줄, 사용자 확인 없음, 단일 사례에 반응)

### 2.2 ✅ 허용된 변경 (책임자 결단)

다음을 모두 충족해야 변경 가능:
1. **§3 트리거 조건 1개 이상 충족**
2. **데이터 표본 ≥ 30 거래일 또는 100건 이상**
3. **근본 원인 분석 문서 1부 작성** (`docs/handover/threshold_review/<날짜>.md`)
4. **사용자 사전 승인** (텔레그램 또는 직접 보고)
5. **A/B 테스트 또는 Walk-forward 검증** 가능 시 우선 진행
6. **변경 후 KPI 모니터링 ≥ 2주**, 악화 시 즉시 롤백

---

## 3. 트리거 조건 — 언제 이 문서를 펴는가

다음 중 **하나라도** 충족 시 임계값 검토를 시작합니다.

### Trigger A: 손절률 정체

W1~W4 작업 완료 후에도 **손절률이 30% 이상으로 정체**되고, KPI 분석 결과 원인이 시장 데이터 가드로 해결 불가능한 영역(=점수 자체의 부정확성)에 있을 때.

**신호:**
- W4 회고 시 손절률 ≥ 30%
- 차단된 종목의 향후 5일 수익률이 통과 종목과 유사 (게이트 무용)
- BUY 결정 종목 중 평균 confidence가 5.5 이하 (저신뢰 매수 다수)

**대응:** buy_threshold (모의 5→6) 또는 Bear 캘리브레이션 검토.

---

### Trigger B: 매수 표본 고갈

가드들이 너무 강력해 매수 자체가 1주 < 5건으로 떨어짐.

**신호:**
- 주간 매수 건수 < 5
- 차단률 ≥ 80%
- 통계 신뢰도 부족으로 KPI 의미 상실

**대응:** 가드 임계값(EXTENDED_CHANGE_PCT, RVOL_MIN_RATIO 등) 완화. **scorer 점수는 손대지 말 것** — 게이트 완화로 해결되는 문제.

---

### Trigger C: 모의 → 실거래 전환 결정

사용자께서 KIS_SIMULATED_MODE=false 전환을 결정하신 시점.

**자동 적용되는 변경 (코드에 미리 박혀있음):**
- buy_threshold: 5 → 7 (이미 구현됨)
- HITL 자동승인 → 텔레그램 수동 승인 (이미 구현됨)

**추가 검토 항목:**
- Bear 캘리브레이션 정상화 여부 (실거래에서는 Bear 발언권 정상화 권장)
- 단일 포지션 캡 추가 축소 (15% → 10%)
- 일일 신규 진입 한도 축소 (3 → 2)

> ⚠️ **실거래 전환 자체는 후배 단독 결정 절대 금지.** 사용자 직접 명령 + 책임자 동석 + 위 자동 변경의 통합 테스트 후 진행.

---

### Trigger D: 시장 국면 장기 전환

KOSPI200이 **3개월 연속 BEAR** (KODEX200 < MA200) 인 상황에서 시스템이 매수 시도를 계속할 때.

**대응:** 임계값이 아닌 시장 국면별 차별 전략 활성화 (P3 항목).

---

## 4. 변경 절차 — 단계별

### Step 1. 이슈 발견 및 트리거 확인 (Day 0)

`docs/handover/threshold_review/<YYYY-MM-DD>_<주제>.md` 신규 작성:

```markdown
# 임계값 변경 검토 — 2026-MM-DD

## 트리거
- [x] Trigger A (손절률 정체)
- [ ] Trigger B
- [ ] Trigger C
- [ ] Trigger D

## 관찰된 데이터
- 손절률: ___% (W___ 시점)
- 표본 수: ___ 거래일 / ___ 건
- 차단률: ___%
- (기타 KPI)

## 가설
"buy_threshold를 5→6으로 올리면 약신호 매수가 줄어 손절률이 ___% 감소할 것이다."

## 대안 (점수 변경 없이)
- (1) 추가 가드 ___ 도입
- (2) 가드 임계값 강화 ___
- 위 대안이 가설을 더 잘 풀 수 있다면 점수 변경 보류.
```

### Step 2. 사용자 보고 (Day 1)

위 문서를 첨부해 사용자께 보고:
```
[임계값 변경 검토 요청]

발견: (1줄)
근거: 표본 ___거래일, KPI 수치
권고: 변경 / 가드 강화 / 추가 관찰
질문: 진행 승인 요청드립니다.
```

### Step 3. 사용자 승인 후 사전 검증 (Day 2~7)

- 과거 데이터로 walk-forward 시뮬: 새 임계값을 작년 데이터에 적용 시 KPI 어떻게 변하는가
- 가능 시 그림자 모드(shadow mode) 운영: 실 시스템은 기존 임계값으로 동작, 로그에는 새 임계값 결정도 같이 기록 → 1주 후 비교

### Step 4. 적용 (Day 8)

- 변경 1건만 적용 (다른 변경과 합치지 말 것)
- git tag로 이전 상태 보존 (`threshold-pre-<주제>`)
- KPI 리포트에 변경 내역 명시

### Step 5. 모니터링 (Day 9~22, 2주)

- 매일 KPI 확인, 기존 트렌드와 비교
- 악화 시 즉시 롤백 (`git checkout threshold-pre-<주제>`)

### Step 6. 회고 (Day 23)

- `docs/handover/threshold_review/` 에 결과 기록
- 채택 / 롤백 / 추가 관찰 결정
- 사용자께 결과 보고

## 5. Bear 발언권 정상화 — 별도 가이드

사용자 의도("모의 공격적 매수")와 잠재적 문제(약신호 매수 → 손절) 사이의 균형을 위한 **선택적 작업**.

### 5.1 적용 시점

- W6 (2026-06-19) 종료 후 KPI 평가
- 손절률이 W6 시점에도 25% 이상이고, 가드들로 더 줄일 여지가 없을 때
- 또는 Trigger C (실거래 전환 결정)

### 5.2 변경 내용 (제안)

**파일:** `autostock/research/debate.py:67`

**현재:**
```python
"실질적인 리스크가 없다면 낮은 bear_score(1~3)를 부여하세요. "
"치명적 리스크에만 높은 점수(8~10)를 부여하세요."
```

**변경 제안:**
```python
"리스크 평가 가이드:\n"
"  1~3점: 일반적 시장 리스크 (모든 종목이 가지는 수준)\n"
"  4~5점: 단기 5% 손실 시나리오가 1개 이상 존재 (예: 차트 패턴 약화, 거래량 둔화)\n"
"  6~7점: 중기 추세 훼손 가능성 (예: MA20 이탈 임박, 수급 이탈)\n"
"  8~10점: 치명적 리스크 (상장폐지, 횡령, 거품 붕괴 임박)\n"
"각 점수의 근거를 bear_summary 에 명시하세요."
```

### 5.3 효과 예측

- Bear 평균 점수: 2.5 → 4.5 정도로 상승 예상
- bull_score - bear_score >= 1.5 게이트 통과율: 90% → 60%로 감소 예상
- 결과적으로 BUY 결정 약 30% 감소 — 약신호 컷이 주효과
- 손절률 추가 감소 기대

### 5.4 모의투자에서의 trade-off

- ❌ 데이터 수집량 감소 — BUY 표본이 줄어 백테스트 자료가 적어짐
- ✅ 손절률 감소 — 자산 보호
- ✅ 실거래 전환 시 충격 흡수 — 모의에서 보수적이면 실거래 전환이 자연스러움

> 💡 **사용자께 직접 의사결정 위임:** "모의 데이터 수집량 vs 손절률 감소" 두 가치 중 어느 것을 우선할지 사용자께서 결정. 후배가 단독으로 결정 금지.

## 6. 모의 → 실거래 전환 시 자동 강화 정책

KIS_SIMULATED_MODE=false 전환 시 다음이 **자동으로** 적용되어야 합니다. 후배는 이 자동 강화가 제대로 동작하는지 코드 점검만 하면 됩니다.

### 6.1 이미 코드에 있는 자동 강화

**파일:** `autostock/research/supervisor.py:51`
```python
buy_threshold = 5 if config.KIS_SIMULATED_MODE else 7  # ✅ 이미 자동
```

**파일:** `autostock/scheduler/jobs.py:136`
```python
if config.KIS_SIMULATED_MODE:
    auto_qty = {...}
    hitl_state.resolve(thread_id, "approved", auto_qty)  # ✅ 모의만 자동승인
else:
    # 텔레그램 HITL 대기 — ✅ 실거래는 사용자 승인 필수
```

### 6.2 추가로 권장하는 자동 강화 (W5 작업 시 함께 구현 권장)


### 6.3 전환 전 통합 테스트 체크리스트

- [ ] 위 모든 환경 분기가 KIS_SIMULATED_MODE=false 시 정확히 동작

## 7. 절대 하지 말 것 (No-Go List)

다음은 **어떤 트리거가 발동해도 금지**:

1. ❌ 사용자 승인 없이 임계값 변경 후 git push
2. ❌ "잠깐 테스트하려고" 임계값 변경 후 원복 잊음
3. ❌ KIS_SIMULATED_MODE=false 단독 결정
4. ❌ 임계값 변경 + 다른 변경(가드 등)을 같은 commit에 묶기
5. ❌ 표본 < 30거래일에서 임계값 변경

> "후배 여러분, 사용자께서 우리에게 '과감한 수정'을 허락하신 것은 **신뢰**의 표시입니다. 그 신뢰는 절차를 지킬 때만 유지됩니다. 즉흥적으로 임계값을 만지는 것은 신뢰의 위반이고, 절차를 거쳐 결단하는 것은 신뢰의 응답입니다."

> "Bear 발언권 정상화는 모의 공격적 매수와 충돌하는 것처럼 보이지만, 실은 **다른 시간 척도의 두 정책**입니다. 모의에서는 데이터 수집이 우선, 실거래에서는 자산 보호가 우선. 우리 코드가 이걸 자동으로 분리해 적용하면 사용자 의도와 안전성을 동시에 만족합니다."

> "마지막으로 — 임계값을 안 만지고 6주를 끝낼 수 있다면 그게 가장 좋은 결과입니다. **시장 데이터 가드로 풀 수 있는 문제는 점수로 풀지 않습니다.** 9개의 안전망(W1~W6)이 모두 작동하는데도 손절률이 높다면, 그때가 이 문서를 펼 때입니다."

— 책임자
